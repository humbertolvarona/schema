1. Nomnoml Fundamentals

1.1 Core Concepts

Nomnoml transforms text definitions into UML diagrams using a simple but powerful syntax:

Basic Structure

// Comments start with //
[NodeName]
[ TypeName| attributes; --; methods]
[Node1] -> [Node2] : relationship

Text Definition

[ User| 
  +username: string; 
  +email: string;
  --
  +login();
  +logout()
]
[User] -> [Session]

Rendered Diagram

[UML Class Diagram visualization]

Key Advantages

  • Text-Based: Version control friendly diagrams as code
  • Rapid Prototyping: Create diagrams faster than drag-and-drop tools
  • UML Compliant: Supports standard UML notation
  • Customizable: Full control over styling and layout
  • Embeddable: Works in documentation, wikis, and web apps

2. Node Types and Classifiers

2.1 Standard Classifiers

Classifier Syntax Appearance Use Case
Class [<class> Name] Rectangle with solid border Standard UML class
Abstract [<abstract> Name] Italicized header Abstract classes
Interface [<interface> Name] Header with «interface» or styled differently Interfaces
Actor [<actor> Name] Stick figure Use case actors
Note [<note> Text] Rounded rectangle Annotations
Package [<package> Name] Folder-style shape Namespace grouping
Database [<database> Name] Cylinder shape Database entities

2.2 Node Structure

Full Node Syntax

[<classifier> Name|
  +publicAttribute: type;
  -privateAttribute: type;
  #protectedAttribute: type;
  --
  +publicMethod();
  -privateMethod();
  #protectedMethod()
]

Advanced Node Features

Multi-line Nodes
[ Product|
  +id: string;
  +name: string;
  +price: float;
  --
  +calculateDiscount();
  +applyPromoCode(code)
]
Static Members
[ MathUtils|
  --
  +{static} PI: float;
  +{static} calculateArea()
]

3. Relationships and Connectors

3.1 Relationship Types

Relationship Syntax Arrow UML Equivalent
Association [A] - [B] ─── Simple association
Directed Association [A] -> [B] ───▶ Directed association
Bidirectional Association [A] <-> [B] ◀───▶ Bidirectional association
Inheritance [A] --|> [B] ───▷ Generalization
Interface Realization [A] ..|> [B] ╍╍╍▷ Interface realization
Aggregation [A] o- [B] ───◇ Shared aggregation
Composition [A] +- [B] ───◆ Composite aggregation
Dependency [A] <.. [B] ╌╌╌▶ Dependency

Labeled Relationships

Multiplicities

4. Directives and Styling

4.1 Core Directives

Layout Directives

  • #direction: right|down|left|up
  • #padding: 10 - Diagram padding
  • #spacing: 40 - Space between nodes
  • #gutter: 20 - Space between lines
  • #zoom: 1.0 - Scaling factor
  • #align: left|center|right

Style Directives

  • #stroke: #333 - Border color
  • #fill: #fff - Background color
  • #fill: #fff -> #eee - Gradient
  • #lineWidth: 1 - Border thickness
  • #arrowSize: 1 - Arrowhead size
  • #edges: hard|rounded

4.2 Typography Directives

Font Directives

  • #font: Helvetica
  • #fontSize: 14
  • #bold: true|false
  • #italic: true|false
  • #underline: true|false
  • #fontColor: #333

Advanced Typography

  • #title: Diagram Title
  • #titleFontSize: 18

Advanced Integration

Dynamic Diagram Generation
// Generate nomnoml code from data
function generateClassDiagram(classes) {
    let code = '#direction: down\n';
    classes.forEach(cls => {
        code += `[ ${cls.name}|\n`;
        code += cls.attributes.map(attr => `  ${attr.visibility}${attr.name}`).join(';\n');
        code += '\n]\n';
    });
    return code;
}

9. Troubleshooting and Best Practices

9.1 Common Errors

Error Solution
Unbalanced brackets Check all [ ] pairs
Missing separators Use ; between members and -- between sections
Invalid classifiers Use only supported types: <class>, <interface>, etc.
Incorrect relationship syntax Use proper arrows: ->, --|>, etc.
Directives not working Place directives at the top, prefixed with #

9.2 Professional Best Practices

  • Use consistent naming conventions (PascalCase for classes)
  • Limit diagrams to 10-15 nodes for readability
  • Use notes and grouping to organize large diagrams
  • Define a standard style guide for your project's diagrams
  • Version control your nomnoml source files
  • Generate complex diagrams programmatically when possible
